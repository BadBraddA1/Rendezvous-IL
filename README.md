# Rendezvous IL Website

Production site for the Rendezvous Christian Homeschool Family Retreat — [rendezvousil.com](https://rendezvousil.com).

Handles public event pages, family registration (2027), admin dashboard (registrations, meals, check-in, messaging), and Clerk-based family accounts.

## Tech stack

- **Next.js 16** (App Router) + React 19 + TypeScript
- **Turso** (libSQL / SQLite) — all app data via `lib/db.ts` and `@libsql/client`
- **Clerk** — authentication (families + admin roles)
- **Resend** — email
- **Vercel** — hosting (project: `v0-ren`)

## Database

- Schema: `scripts/schema-turso.sql`
- All queries: `import { sql } from "@/lib/db"`
- Migrated from Neon Postgres to Turso (one-time script archived in `scripts/archive/`)
- Key tables: `families`, `family_members_v2`, `registrations`, `pending_family_changes`

## Family accounts (`/account/profile`)

- Clerk users are linked to a `families` row by `clerk_user_id` or matching registration email (auto-linked on first visit).
- Profile edits do **not** write directly to Turso — they queue rows in `pending_family_changes` for admin approval at `/admin/pending-changes`.
- Member type / age group: adults and teens skip the age-group picker; children with a registration birthday get age group auto-calculated (`lib/member-age.ts`).
- Admin roles: `admin`, `editor`, `viewer`, and **`checkin`** (check-in station only — web + iOS). Permissions live in `lib/clerk-auth.ts` (`getAdminPermissions`).
- **`/admin/users`** — full user management (Clerk): create/delete users, assign roles, ban/unban, reset passwords, and track **last seen + platform** (`web` / `ios` / `android`) via `user_app_activity` and `POST /api/auth/activity`. Web admin UI at `/admin/users`; iOS **More → User management** (full admins only).
- **`/admin/audit`** — expandable activity feed: search, date range, action filters, before/after field changes, and summary stats (`GET /api/admin/audit?action=&from=&limit=`).
- **`/admin/displays`** — Raspberry Pi Live Updates fleet: last seen, current view, online/stale/offline (`live_updates_displays`, `POST /api/live-updates/heartbeat`, `GET /api/admin/displays`).
- **`/admin/photoshow`** (nav: Communication → Photoshow) — slideshow photos for Live Updates and room TVs. Upload JPG/PNG/WebP (Vercel Blob under `photoshow/`), optional captions, reorder, show/hide. Active photos join the Live Updates rotation automatically; dedicated room mode (replaces laptop slideshows) is `/live-updates?kiosk=1&view=photoshow` (full-bleed, no program chrome). All screens stay on the same slide (8s, epoch-aligned). Public list: `GET /api/live-updates/photos`; admin: `GET/POST /api/admin/photoshow`, `PATCH/DELETE /api/admin/photoshow/[id]`. Table: `live_updates_photos` (created lazily).
- **Registration previews** (`/admin`) — toggles for admin test registration (`/registration-test2026`) and express registration preview (`/account/express-registration`); stored in `app_settings`.
- **Express registration (review & confirm)** (`/account/express-registration`, behind the preview toggle) — a fast path for returning families. `lib/express-registration-prefill.ts` loads the family's most recent prior-year registration (registration row + family members + t-shirt orders + health info + volunteer signups + session suggestions) and maps it into the standard `RegistrationData` shape with **ages recomputed for the 2027 event date** (from DOB when stored, else stored age + year gap). The client (`express-registration-client.tsx`) walks 6 steps — Family Info, Lodging, Merchandise & Add-ons, Additional Info, Agreement, Review & Submit. Steps 1–2 (Family Info, Lodging) show a read-only summary of last year's answers with a "this is what you had for {year} — still right?" banner and two buttons: **Yes, still right — continue** or **Make changes** (which swaps in the real form step components, so edits use the exact same UI/validation as the full form). Steps 3–4 are **fresh every year** — t-shirts, climbing tower, scholarship fund, emergency contact, and worship-service volunteers are never prefilled (health info and session suggestions do carry over into the step-4 editor); step 4 requires an emergency contact name + phone before continuing. Signatures never carry over; the Agreement step must be re-signed. Submission posts to `POST /api/express-registration/submit` (gated on the express preview toggle, not the test-registration toggle) which reuses `submitTestRegistration` with `payment_notes = 'EXPRESS_TEST'` and fires signature-request emails when that feature is on. Lodging totals recompute automatically when members change (shared math in `lib/lodging-cost.ts`, also used by the Lodging step). If no prior registration exists for the family's email, the page points to the standard form. The admin dashboard "Express registrations" stat counts `registrations` rows with `payment_notes = 'EXPRESS_TEST'` for 2027 (the old `express_registration_2027` preferences API still exists but the client no longer writes to it).
- **Family Directory editor** (`/admin/directory`, nav: Registrations → Family Directory) — admins can edit every family's directory card per event year (year picker matches the other admin pages). Lists all families with a registration for the year **including ones hidden from the public directory** (badge shows "Hidden from directory"). Editable per family: last name, husband/wife first names, home congregation, address/city/state/zip, 280-char blurb, public listing toggle, and the family photo (upload/replace/remove, reuses the same Blob storage as the self-service uploader). Each card also lists the registration members that drive the public member list, with per-member editing: name, age, Father/Mother role, email, phone, and the "share contact info in directory" flag. Search filters by family, member, congregation, or email. **Bulk edit** — a toggle makes every family and member field on the page editable at once; changes accumulate locally (edited cards get a highlight + "Edited" badge) and a sticky bottom bar shows the unsaved count with **Save all** (single `POST /api/admin/directory/bulk`, only changed fields are sent; per-item failures are reported without aborting the batch, one `bulk_update_directory` audit entry) and **Discard**. Photo uploads stay immediate and per-card. API: `GET /api/admin/directory/families?year=`, `PATCH /api/admin/directory/families/[id]`, `POST/DELETE .../[id]/photo`, `PATCH /api/admin/directory/members/[id]`, `POST /api/admin/directory/bulk` — writes require editor/admin role and are audit-logged (`update_directory_family`, `update_directory_family_photo`, `update_directory_member`, `bulk_update_directory`). Data lives on `families` (card fields) and `family_members` (member rows from the year's latest registration per email) — the same sources `lib/family-directory.ts` reads, so edits show up in the public `/directory` immediately.
- **T-shirt size chart** — the Merchandise step's Size Chart button opens a native HTML dialog (`components/registration/size-chart-dialog.tsx`) with Adult/Women's/Youth/Toddler/Infant measurement tables transcribed from the supplier PDF (scoped to the sizes we sell), replacing the old `shirt-size-charts.png` image popup.
- **Registration member contacts** — each family member on the registration form has optional `email` / `phone` fields (stored on `family_members`; migration: `scripts/add-family-member-contact.sql`). Email is **required** for members with the Father or Mother role, and the two parents must use **different** email addresses (enforced inline in the form and again on the step-1 Next button). Phones are always optional and normalized on blur via `lib/phone-format.ts`.
- **Event schedule editor** (`/admin/schedule`, nav: Event → Schedule) — the public daily schedule is now DB-backed and admin-editable. Rows live in `schedule_events` (per event year; created lazily by `lib/event-schedule.ts`) with day, time, title, location text, note, plus the interactive extras: **map pin** (venue-map location id → tappable `LocationLink`), **meal menu** (breakfast/lunch/dinner → inline `MealMenu`), **worship volunteer slot** (Morning/Evening Devotion → inline `VolunteerSchedule`), **info link** (e.g. `/scrabble`, `/biblebowl`), and a **weather chip** flag for outdoor events. Full CRUD grouped by day with move up/down reordering (`POST /api/admin/schedule/reorder`); same shared year picker as the other admin pages. **Fallback & seeding**: while a year has no rows, every consumer renders the built-in schedule from `lib/schedule-data.ts` enriched with the same metadata (heuristics in `staticPublicDays`), and the editor offers an **"Import the built-in schedule"** button (`POST /api/admin/schedule/seed`, no-op if rows exist) so nobody types 60 events by hand. **One source of truth** (`getPublicSchedule`): `/schedule` (rebuilt as a data-driven server page + `schedule-client.tsx` renderer), `/schedule/print`, the PDF (`/api/schedule/pdf`), the apps' JSON (`/api/schedule`, same shape as before plus the extras), and Live Updates — the TV shell fetches `luItems` from `/api/schedule` every 10 minutes (static list as fallback) so mid-event edits reach the TVs without a redeploy, and the server-side volunteer card (`lib/live-updates/server-volunteer.ts`) finds the next assembly from the edited schedule too. API: `GET/POST /api/admin/schedule`, `PATCH/DELETE /api/admin/schedule/[id]` (all audit-logged; editing requires admin/editor role).
- **Volunteers** is its own top-level nav group with three pages sharing one auth shell (`app/admin/volunteers/volunteer-admin-shell.tsx`) and one year-picker wrapper (`components/admin/volunteer-section.tsx`): **Worship Service** (`/admin/volunteers`), **Lesson Bids** (`/admin/volunteers/lesson-bids`), and **Special Assignments** (`/admin/volunteers/special-assignments`).
- **Special assignments** (`/admin/volunteers/special-assignments`) — one-off jobs outside the devotion rotation (activity sessions, lifeguards, check-in crew, etc.), rebuilt on the old dash's surviving `special_assignments` table (lazy `event_year` column, NULL = 2026). Fields: activity, person (optional — blank shows an "Unassigned" badge so you can create the slot first and book someone after they register), date, time/session, notes; full CRUD, per-year, audit-logged. When a year is empty, a **"Load last year's activity list"** button seeds it with the 2026 activity slots (`STARTER_ACTIVITIES` in `lib/special-assignments.ts`, dates computed from that year's event Monday, nobody booked) — refuses to run if the year already has rows. API: `GET/POST /api/admin/special-assignments`, `PATCH/DELETE /api/admin/special-assignments/[id]`, `POST /api/admin/special-assignments/seed`.
- **Worship service volunteers** (`/admin/volunteers`, nav: Volunteers → Worship Service) — manages the "Volunteer for Worship Services" signups collected during registration (`volunteer_signups`). An **event year picker** (same `?year=` + sessionStorage pattern as Registration Management, shared storage key so the year follows you between admin pages) filters everything by the registration's `event_year` and switches the schedule editor's service dates (Mondays per year in `EVENT_MONDAY`, `components/admin/volunteer-schedule-manager.tsx`). The page shows the **Schedule** (pick one of the 8 devotion services for the selected year, e.g. Mon 5/3 PM – Fri 5/7 AM for 2027, and fill its 8 slots — Opening/Closing Prayer, A/B Leading singing, A/B Reading scripture, A/B Presenting a lesson with title + scripture fields — plus a roster to search/filter all signups, see or clear assignments, jump to the family's registration). Slots can be filled three ways: the **dropdown**, **drag-and-drop** (drag a chip from the "Available for [service]" tray onto a slot — slots only accept matching roles), or **Auto-fill week** (greedy fill of every empty slot across the week that balances load and prefers topic-winners for lesson slots; shows a **draft preview grouped by service that saves nothing until "Approve & apply"**, which posts to `POST /api/admin/volunteers/bulk-assign`). **Booking rules** (enforced in the UI and again server-side in `lib/volunteer-scheduling.ts`, returning 409): one volunteer per slot, and one booking per person per day — a person = registration + name, so someone signed up for several roles still gets at most one slot per day and each role at most once. Assigning writes `assigned_date` / `time_slot` / `prayer_type` on the signup, which the existing public schedule (`GET /api/volunteer-schedule`, `/schedule` page, Live Updates) reads directly — no schema change. **Volunteer contact emails**: the registration form's volunteer section asks for an email per volunteer (pre-filled from the member's email; stored as `volunteer_signups.volunteer_email`, column added lazily), shown in the roster. **Email assignments** button sends each assigned volunteer one email listing all their services/roles (+ lesson title/scripture for presenters) via `POST /api/admin/volunteers/send-assignments`, stamping `schedule_email_sent_at`; recipients resolve volunteer email → matching family member email → family primary email (shared logic: `lib/volunteer-scheduling.ts` `resolveVolunteerEmail`, also used for lesson-bid invites). API: `GET /api/admin/volunteers`, `PATCH /api/admin/volunteers/[id]`, `POST /api/admin/volunteers/bulk-assign`, `POST /api/admin/volunteers/send-assignments` (all audit-logged; editing requires admin/editor role, viewers are read-only).
- **Lesson bid system** (`/admin/volunteers/lesson-bids`, nav: Volunteers → Lesson Bids) — rebuilt from the old admin dash on the surviving `lesson_topics` + `lesson_bids` tables (created lazily by `lib/lesson-bids.ts`). Topics are **per event year** (`lesson_topics.event_year`, added lazily, NULL = **2026** so last year's topics stay under Rendezvous 2026) and the page follows the shared year picker. Each topic has **three fields**: topic (the theme, column `title`), suggested lesson title (`lesson_title`), and scripture (`scripture`) — the schedule shows `presenter title → topic's lesson title → topic name` and `presenter scripture → topic scripture`. Admins maintain the **topics list** (add/edit/delete; claimed topics can't be deleted) and see every "Presenting a lesson" volunteer with a status chain: Not invited → Invited → Bid submitted (ranked picks shown) → Awarded. **Send bid invite** emails the presenter a personal `/lesson-bid/[token]` link (Resend; recipient = the family member whose first name matches the volunteer and has an email, else the registration's primary email; token stored on `volunteer_signups.lesson_bid_token`). Topics are **first come, first served**: the public page shows open topics and the presenter claims exactly one (`claimTopicByToken` uses a guarded `UPDATE … WHERE claimed_by_volunteer_id IS NULL` so a race returns 409 "someone else just claimed that" instead of double-booking). **After claiming, the same link stays live** so the presenter can enter their lesson title and scripture reading (`volunteer_signups.lesson_title` / `scripture_reading`), which **override the raw topic title** on the public schedule (`COALESCE(NULLIF(vs.lesson_title,''), lt.title)`). Admins can still **assign a topic manually** or **release** a claim (the old award/unaward endpoints, audit-logged) — e.g. for presenters without email. Claiming sets `volunteer_signups.claimed_lesson_id` + `lesson_topics.claimed_*`, so the topic title shows on `/schedule` and Live Updates immediately. (The original ranked-picks flow was replaced; `lesson_bids.pick_1..3` columns remain in the DB but are unused.) The Schedule tab's lesson slots show a read-only "Awarded topic" hint instead of the manual fields when a topic is claimed. API: `GET/POST /api/admin/lesson-topics`, `PATCH/DELETE /api/admin/lesson-topics/[id]`, `POST /api/admin/volunteers/[id]/lesson-bid` (`invite` / `award` / `unaward`), public `POST /api/lesson-bid/[token]`.
- **Parent signature emails** (toggle: "Parent signature emails" on `/admin`, `app_settings.signature_emails_enabled`, default **off**) — when on, the Agreement step's typed signature boxes are replaced with a notice, and after submission each parent gets a Resend email with a personal `/sign/[token]` link (token = 64-char hex, no expiry). The signing page shows the agreement text (shared source: `lib/agreement-content.ts`), records the typed signature, and then shows whether the other parent has signed. Data lives in `signature_requests` (created lazily by `lib/signature-requests.ts`; script: `scripts/create-signature-requests-table.sql`). Admin: registrations list shows a red "Signatures x/y" badge, and the registration edit page has a Parent signatures panel with **Resend email** and **Mark signed** (audit-logged). **Check-in is blocked (409)** while signatures are pending *and* the toggle is on — turning the toggle off removes the gate entirely. Families with no Father/Mother roles get a single signing link at the primary email.
- **Rate calculator** (`/calculator`, `/admin/calculator`, `/admin/rates`) — lodging line items are **per person**; RV/tent **site fees** use one-tap night buttons (per site). Users pick **nights and meals** (custom schedule picker — no package dropdown); the server detects full week, 3/9, 2/6, or 1/3 packages and prices accordingly (`lib/calculator-schedule.ts`, `lib/calculator-estimate.ts`, `POST /api/calculator/estimate`). Admin calculator and signed-in family estimate share the same detection logic. **Guests** on `/calculator` see a sign-in / create-account prompt explaining that returning families get a pre-filled estimate. **Signed-in families** with a prior registration are **auto-loaded** from last year (`/api/calculator/family`, `lib/calculator-family-seed.ts`, `components/calculator/family-estimate.tsx`) — members, nights, meals, lodging, and a year-over-year total comparison — without a manual “load” step. The quick calculator enters **family members per person** (optional name + the registration form's birth-year `CustomDateSelector` / 18+ checkbox) and buckets them into adult/youth/child/infant pricing by age on the event date; signed-in users get a **"Save for registration"** button that stores members + lodging (`calculator_prefill` table keyed by Clerk user + year, `lib/calculator-prefill.ts`, `GET/POST /api/calculator/prefill`), and the registration form **auto-populates** family members (and rv/tent lodging) from that save on first load — only while the form is still untouched, with a notice to double-check.
- API: `GET/PUT /api/family/profile`, `POST/DELETE /api/family/members`

## Family directory (`/directory`)

- Registered families can upload a photo and short blurb from `/account/profile`; listings appear at `/directory` for the same event year.
- **Listing is on by default** for registered families (`directory_opt_in`); families opt out from profile if they do not want to appear. Photos are optional — families without a photo still show name, congregation, contact info, and attendees.
- **Phone numbers** are stored on each `family_members_v2` row (`phone` column) and shown in the directory with that member's name (not husband/wife guesses). Legacy family-level phones still display if no member phones are set. US numbers are normalized to `(XXX) XXX-XXXX` on save (`lib/phone-format.ts`); retroactive cleanup: `pnpm db:normalize-phones`.
- **Structured member listings + per-member contact opt-in** — directory cards call out **Father:** / **Mother:** by name and list **Kids:** oldest-to-youngest with ages ("Adult" for 18+ or `is_adult_override`), sourced from the year's registration `family_members` rows (`parent_role` column; latest registration per family email). Names and ages always show; a member's **email/phone only appears if the "Show this email/phone in the Family Directory" checkbox** was ticked for them on the registration form (`family_members.share_contact_directory`, default off — both columns added lazily). Opted-in registration phones take precedence over profile (`family_members_v2`) phones; families with no registration member rows fall back to the legacy husband & wife names + profile phones display.
- **Year visibility** is controlled on the admin dashboard (`/admin`) — toggles per year in `app_settings` (`directory_enabled_2026`, `directory_enabled_2027`). Defaults: **2026 on**, **2027 off** until registration opens.
- Public API: `GET /api/directory/years` returns enabled years; `GET /api/directory?year=` requires sign-in + registration for that year (admins can preview disabled years).
- Admin API: `GET/POST /api/admin/directory/status` (POST requires `admin` role).
- iOS: `DirectoryView` and `FamilyDirectoryManageView` in `ios/RendezvousIL/`.
- **`/map2026` (Attendee)** merges the live directory with geocoded map pins when signed in (`GET /api/map2026/attendees?year=`). Password-only access falls back to archived static pins until sign-in.

## Year chat (`/chat`)

- **Core attendee feature** — each event year gets a group chat (`year-2026`, `year-2027`). Register for a year → you can message that year's families; past years you attended stay available.
- **Realtime** via Ably (`rendezvous:channel:{id}`); token at `POST /api/ably/token`. Tables: `chat_channels`, `chat_channel_members`, `chat_messages` (lazy-created by `lib/chat-schema.ts`).
- **Member UI:** `/chat` (web) and iOS **Chat** tab (`ChatListView` / `ChatThreadView`). Requires Clerk sign-in + registration for year channels. Web uses Ably for live messages (polls as fallback). Members can send photos (Blob storage). New messages send APNs/FCM to other channel members (device tokens linked to Clerk user id).
- **Mods:** Site admins moderate all rooms. Channel **moderators** (set in Admin → Year Chat) can delete messages and post announcements only in rooms they moderate.
- **Admin:** `/admin/chat` (nav: Communication → Year Chat) — create **test/custom** channels, pick members, **Make mod / Remove mod**, chat from the web. Year channels auto-include registered families.
- API: `GET /api/chat/channels`, `GET/POST /api/chat/channels/[id]/messages`, `DELETE /api/chat/messages/[id]`; admin `GET/POST/PATCH/DELETE /api/admin/chat/channels`, `GET /api/admin/chat/people`, `GET/POST/DELETE /api/admin/chat/channels/[id]/members`.

## iOS app

Native **SwiftUI** attendee hub in `ios/` (not a WebView shell). **Sign-in required** for almost everything — welcome screen, then Clerk, then tabs:

- **Home** — greeting, retreat shortcuts (schedule, updates, chat, directory)
- **Schedule** — week view, meals, worship leaders, event reminders
- **Updates** — now/next, weather, announcements
- **Chat** — year group chat (Ably)
- **More** — directory, account, Bible Bowl, FAQ, notifications, admin/check-in for staff

Setup: `cd ios && bash scripts/setup-xcode.sh && open RendezvousIL.xcodeproj` — see [ios/README.md](ios/README.md). For staff sign-in, add your Clerk key to `ios/Config.local.xcconfig`.

**TestFlight:** `cd ios && bash scripts/ship-testflight.sh` — latest **v1.2.0 (build 14)**. Device smoke checklist: [ios/TESTFLIGHT_SMOKE.md](ios/TESTFLIGHT_SMOKE.md).

Native admin APIs: `GET /api/admin/me`, `GET /api/admin/mobile/dashboard` (Bearer Clerk session token). Mobile Bearer auth always falls back to `authenticateRequest` / `verifyToken` using the `Authorization` header from the request or Next.js `headers()` — so admin routes that omit the `Request` argument no longer fail intermittently on first open.

**App Store listing:** description, keywords, Marketing/Support/Privacy URLs, and review notes are in [`ios/app-store/metadata.md`](ios/app-store/metadata.md). Screenshots: `bash ios/scripts/capture-app-store-screenshots.sh` → `ios/AppStoreScreenshots/`.

## Android app

Native **Jetpack Compose** app in `android/` (Phases 1–4):

- **Home / Schedule / Updates / More** — 4-tab shell matching iOS
- **Schedule** — day picker, meals, volunteer slots, **event reminders** (bell icon), offline fallback
- **Updates** — now/next (Central Time), weather, announcements
- **More** — calculator, Bible Bowl, FAQ, About, **Clerk account**, **directory**, **notifications & widgets**, **admin** (dashboard, user management, staff check-in)
- **Admin** — mobile dashboard stats, staff check-in with QR scan, user CRUD (role-gated via Clerk session)
- **Push** — FCM organizer broadcasts; `POST /api/push/register` with `platform: "android"`
- **Widgets** — Glance home screen: Next event + Now & next
- **Deep links** — `rendezvousil://schedule` opens Schedule tab

Setup: open `android/` in Android Studio (JDK 17), copy `local.properties.example` → `local.properties`, add `CLERK_PUBLISHABLE_KEY`, add `google-services.json` (or Firebase manual keys) for FCM — see [android/README.md](android/README.md) and [docs/FCM_SETUP.md](docs/FCM_SETUP.md).
